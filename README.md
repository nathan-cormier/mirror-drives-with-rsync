# mirror-drives-with-rsync

An interactive CLI tool for mirroring subdirectories from a source root to a destination root using `rsync`. It lists the available subdirectories, lets you choose which ones to sync, and handles resuming interrupted syncs.

Apple sidecar files (`._*`) generated by macOS on exFAT/FAT32 volumes are excluded from all sync and comparison operations.

## Requirements

- Go 1.21+
- `rsync` (version 3.0+ recommended — install via `brew install rsync` on macOS)

## Installation

```sh
go install github.com/nathan-cormier/mirror-drives-with-rsync@latest
```

Or build from source:

```sh
go build ./cmd/mirror-drives-with-rsync
```

## Usage

```
mirror-drives-with-rsync [-dry-run] [-resume] [-compare] [-no-delete] <source_root> <dest_root>
```

### Flags

| Flag | Description |
|------|-------------|
| `-dry-run` | Preview what rsync would do without making any changes (uses `rsync -n -i`) |
| `-resume` | Continue a previously interrupted sync using saved state |
| `-compare` | Print file counts and total sizes for both roots, plus a diff of files unique to each side |
| `-no-delete` | Copy files from source to destination without deleting extraneous files on the destination (omits rsync's `--delete`) |

## Examples

### Normal sync

```
$ mirror-drives-with-rsync /Volumes/Camera /Volumes/Backup

rsync version 3.3.0  protocol version 33

Subdirectories on source: /Volumes/Camera
Destination root: /Volumes/Backup

   1) 2022
   2) 2023
   3) 2024

   *) 2019  (exists only on destination)

Enter numbers to mirror (space or comma separated), or "all" for every subdirectory.
Selection: 2 3

Will sync:
  /Volumes/Camera/2023 -> /Volumes/Backup/2023
  /Volumes/Camera/2024 -> /Volumes/Backup/2024
Proceed? [y/N] y

>>> rsync: /Volumes/Camera/2023/  ->  /Volumes/Backup/2023/
...
>>> rsync: /Volumes/Camera/2024/  ->  /Volumes/Backup/2024/
...

Done.
```

Directories shown in yellow with `*` exist on the destination but not the source — they are informational only and cannot be selected.

### Sync all subdirectories

```
Selection: all
```

### Dry run

Preview what would be transferred without making any changes:

```
$ mirror-drives-with-rsync -dry-run /Volumes/Camera /Volumes/Backup

rsync version 3.3.0  protocol version 33
Mode: dry run (rsync -n -i)

Subdirectories on source: /Volumes/Camera
Destination root: /Volumes/Backup

   1) 2022
   2) 2023
   3) 2024

Enter numbers to mirror (space or comma separated), or "all" for every subdirectory.
Selection: 3

Will sync:
  /Volumes/Camera/2024 -> /Volumes/Backup/2024
Proceed? [y/N] y

>>> rsync [dry-run]: /Volumes/Camera/2024/  ->  /Volumes/Backup/2024/
>f+++++++++ IMG_0001.JPG
>f+++++++++ IMG_0002.JPG
...

Done. (dry run — no changes made)
```

### Keep extraneous destination files (no delete)

By default the tool runs a true mirror: files on the destination that no longer exist on the source are deleted (rsync's `--delete`). Use `-no-delete` to make the sync additive instead — files are copied from source to destination, but anything already on the destination is left untouched:

```
$ mirror-drives-with-rsync -no-delete /Volumes/Camera /Volumes/Backup

rsync version 3.3.0  protocol version 33
Mode: no-delete (extraneous files on dest will be kept)

Subdirectories on source: /Volumes/Camera
Destination root: /Volumes/Backup

   1) 2022
   2) 2023
   3) 2024

Enter numbers to mirror (space or comma separated), or "all" for every subdirectory.
Selection: all

>>> rsync: /Volumes/Camera/2024/  ->  /Volumes/Backup/2024/
...

Done.
```

This flag can be combined with `-dry-run` to preview an additive sync.

### Resume an interrupted sync

If a sync was interrupted, resume from where it left off:

```
$ mirror-drives-with-rsync -resume /Volumes/Camera /Volumes/Backup

rsync version 3.3.0  protocol version 33
Mode: resume

I already synced these folders: 2022, 2023
I will resume with these folders: 2024
Continue? [y/N] y

>>> rsync: /Volumes/Camera/2024/  ->  /Volumes/Backup/2024/
...

Done.
```

State is saved in `.mirror-drives-with-rsync-state.json` at both the source and destination roots after each folder completes. If both state files exist and differ, the tool will error rather than guess which to trust.

### Compare source and destination

Check whether a sync is complete without running rsync:

```
$ mirror-drives-with-rsync -compare /Volumes/Camera /Volumes/Backup

Scanning source: /Volumes/Camera
..........
Scanning dest:   /Volumes/Backup
.......

Source: 12345 files, 42.17 GiB
Dest:   12300 files, 41.98 GiB

Only in source (45):
  2024/IMG_0041.JPG
  2024/IMG_0042.JPG
  ...
Only in dest (2):
  2019/IMG_9999.JPG
  2020/IMG_0001.JPG
```

If source and destination are identical:

```
Source: 12345 files, 42.17 GiB
Dest:   12345 files, 42.17 GiB

No differences.
```

Progress dots are printed as directories are scanned (one dot per 10 directories).

## State file

During a normal (non-dry-run) sync, `mirror-drives-with-rsync` writes `.mirror-rsync-state.json` to both the source and destination roots. This file records which folders were selected and how far the sync has progressed, enabling `-resume` after an interruption.

The state file is kept after a successful run (with `nextIndex` equal to the total folder count), so `-resume` can detect that the previous run completed cleanly.
